# Skal — open items

Real items that have been deferred, in roughly the order of priority. Not a
wishlist — only things we've actually hit and chosen to defer.

## Build pipeline

### Use vendored bun (not host bun) for bytecode generation
`scripts/build-app.sh` and `js-app/package.json` invoke `bun` via `$PATH` to
produce `skal-app.cjs.jsc`. That's the user's globally installed bun
(currently 1.3.13), not `vendor/bun/` (also at 1.3.13 — same commit by
coincidence). When they drift, JSC silently rejects the bytecode at runtime
and falls back to parsing — observable only as a perf regression, no error.

Fix: build `vendor/bun/` for darwin host, store the binary in
`build/host/bun`, point `build:bytecode` at it. ~1 hr to wire, ~30 min to
actually build host bun on first run.

See `docs/bytecode-cache.md` § "JSC version coupling".

### gitignore JS build artifacts
`android-app/app/src/main/assets/skal-app.js`, `skal-app.cjs`, and
`skal-app.cjs.jsc` are build outputs (regenerated by `scripts/build-app.sh`).
They're currently tracked in git, which causes diff noise on every build
because `.cjs.jsc` is non-deterministic across bun runs.

Fix: add to `.gitignore`, `git rm --cached` them. Confirm
`scripts/build-app.sh` always builds them before APK assembly so a clean
checkout still produces a working APK.

### Bytecode is non-deterministic across bun runs
Two consecutive `bun build --bytecode` runs on the same source produce
slightly different `.jsc` bytes (we observed 228K vs 234K). Likely a
timestamp or RNG-seeded ID baked into the bytecode header. Investigate
whether bun has a `--reproducible` flag or env var. If not, this is a
limitation we live with (gitignoring the .jsc anyway makes it moot).

### APK install-time bytecode validation
No mechanism to detect when a libskal.so update has invalidated the shipped
bytecode. JSC silently rejects → fallback to parse. Add a build-time hash
of `getJSCBytecodeCacheVersion()` baked into the bytecode generator, check
at app startup, log a warning if mismatched.

## Runtime / Performance

### Skip asset extraction when APK hasn't changed
`MainActivity.extractAsset()` re-copies `skal-app.cjs` + `.cjs.jsc` from
APK assets to filesDir on every launch (~1–10 ms for 250 KB). Compare APK
last-modified time vs `filesDir/skal-app.cjs.mtime` and skip the copy when
up to date. Easy win; not currently a bottleneck.

### LazyColumn / virtualization for long lists
`<For each={tweetsToShow()}>` with 5000 items renders **all** 5000 in a
non-virtualized scroll Column. Compose has to measure all 5000 every
relayout. We don't have a `WT_LAZY_COLUMN` widget type yet — Solid's
universal renderer model and our bridge don't currently express
"viewport-driven mounting." Worth designing if Skal targets serious list
UIs.

### Hot reload over a debug channel
Debug builds re-parse `skal-app.js` (the IIFE) on every launch — that's
already set up to be hot-reload-friendly. Missing: a debug channel that
pushes updated source to a running runtime, triggers re-evaluation,
preserves Solid signal state across the swap. Likely a small server in
the Android process that watches for `adb push`-driven source updates.

### `dispatchEvent` uses plain `putLong` for `event_seq`
`SkalBridge.kt:dispatchEvent` writes `event_seq` via plain `buffer.putLong`,
which is acquire-release-equivalent on ARM64 by hardware contract. Strict
correctness would use `VarHandle.setRelease`. Same generated code in
practice, pedantically correct.

### `EvalRequest` allocation per click in skal_entry.zig
`nativeWakeJs` allocates a fresh `EventDrainTask` (~64 bytes via mimalloc)
per click. For touch/scroll streams (60–120/sec) it's irrelevant; for
hypothetical high-frequency input (motion sensors at 1 kHz) we'd want
a free list. Pool the tasks if/when this becomes a real load.

## Architecture

### `console.error` crashes in embedded environment
`Bun::ConsoleObject::messageWithTypeAndLevel` segfaults when called from
our worker because Bun's stdio isn't wired up for the embedded case (no
real stderr). Worked around in `bridge.js` by capturing exceptions into a
`lastHandlerError` module global instead of `console.error`. Real fix:
register a console handler in `skal_entry.zig` that bridges to Logcat
via `__android_log_write`. Would also enable `console.log` from JS
to show up in `adb logcat`.

### `OP_REMOVE_NODE` deep recursion
`SkalBridge.removeSubtree` walks the children list recursively. For deeply
nested trees (>1000 levels), this would stack-overflow on the UI thread.
Iterative version with an explicit stack would be safer. Current trees are
shallow (<5 levels), not a real bug today.

### Solid universal-renderer fragment footgun
`App()` returning a fragment `<>...</>` causes Solid's universal renderer
to wrap the children array in a single reactive effect that re-evaluates
every function child on any signal change — exploding into hundreds of
ops per `setCount` call. We documented this in `App.jsx` (with a
load-bearing comment about always wrapping in `<column>`), but it's a real
gotcha for any Skal app developer. Worth addressing in a developer-facing
README when one exists, OR investigating whether babel-preset-solid's
universal mode could emit different code that doesn't have this trap.

## Documentation

### Public README at repo root
There's no README.md at the project root. The codebase has `CLAUDE.md`
(meta-instructions), `docs/bytecode-cache.md` (architecture deep-dive),
`patches/SKAL_WIRE.md` (wire format spec), and component-specific
comments. A high-level "what is Skal, how to build it, how to run the
demo" doc would help anyone arriving cold.

### Onboarding doc for app developers
The `js-app/` directory has no README explaining: how to add new widgets,
the universal-renderer + babel-preset-solid setup, the fragment-vs-element
gotcha, the bridge wire format from a JS-author POV. Different audience
from `docs/bytecode-cache.md` (which is for Skal-internal dev).

### Symbolicating native crashes
We had a SIGSEGV in `Bun::ConsoleObject` and walked through symbolicating
it via `llvm-addr2line` against the unstripped libskal.so. That workflow
should be a doc — anyone hitting a native crash needs the build-id from
logcat → unstripped .so → addr2line incantation.
