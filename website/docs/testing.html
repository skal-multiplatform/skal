Testing

Three runtimes, one test story: fast unit tests in bun test, host tests in flutter test, and Maestro flows tapping the real app on a real device. Here's what to test where — with working examples.

What to test where

LayerRunnerSpeedReach for it when…
Unit / componentbun testmslogic, store behavior, a component's rendered tree, regressions you can express as ops
Host (Dart)flutter testsecondsyou wrote a custom widget adapter or touched wire decoding
E2EMaestro~30 s / flowuser journeys: launch, navigate, tap, type, assert — the whole stack

Rule of thumb: every bug gets a unit test; every user journey gets a flow. The host layer is the framework's job — you only add Dart tests when you ship your own adapters.

Unit & component tests (bun test)

The scaffold is pre-wired: *.test.jsx anywhere under src/ runs with skal test (an alias for bun test). Components render headless — skal/testing installs an in-memory bridge, so your JSX produces the same binary ops it would on a device, without a device:

// src/Counter.test.jsx
import { test, expect } from 'bun:test';
import { render } from 'skal/testing';
import { Counter } from './Counter';

test('increments on tap', () => {
  const app = render(() => <Counter />);

  app.getByTestID('inc').tap();
  app.getByTestID('inc').tap();

  expect(app.getByTestID('count').props.label).toBe('Tapped 2 times');
});

The handle you get back is the mounted tree, queryable the same way Maestro queries the real app — by testID first:

APIWhat it does
render(fn)mounts a component on the in-memory bridge; returns the app handle
getByTestID(id) / queryByTestIDfind a node (throws / returns null)
getByText(str)find by visible label
node.tap() · node.input(str)dispatch real bridge events into your handlers
node.props · node.childrendecoded prop table + subtree, plain objects
app.ops()the raw op log since last drain — assert how little re-rendered
app.unmount()dispose Solid roots (auto-runs per test)
Because Solid is fine-grained, app.ops() makes a great performance regression net: assert a signal update emits one SET_TEXT, not a subtree rebuild. When someone accidentally maps a fresh array through a list, this test catches it in milliseconds.

Testing store logic

In tests the store uses an in-memory backend by default — same API, no disk, fresh per test. Pass a seed to start from a known state:

import { test, expect } from 'bun:test';
import { createTestStore } from 'skal/testing';
import { schema } from './db';

test('migration backfills locale', () => {
  const db = createTestStore(schema, {
    seed:        { settings: { theme: 'dark' } },  // pre-migration shape
    seedVersion: 1,
  });
  expect(db.settings.locale).toBe('en');           // migrate() ran
});

Host tests (flutter test)

Only needed when you write custom Dart adapters. Put widget tests next to your adapter in flutter-host/test/; the framework's own decoder, wire format, and builders are covered upstream. skal test host runs them.

E2E with Maestro

Flows are declarative YAML driving the real app through the platform accessibility tree — you write user journeys in neither of the app's two languages. The scaffold ships .maestro/smoke.yaml; add one file per journey:

# .maestro/checkout.yaml
appId: com.example.myapp
---
- launchApp
- extendedWaitUntil:                 # covers async JS boot — always start with this
    visible: { id: "home-title" }
    timeout: 15000
- tapOn: { id: "product-3" }
- inputText: "2"
- tapOn: { id: "add-to-cart" }
- swipe: { direction: UP }
- tapOn: { id: "checkout" }
- assertVisible: { id: "order-confirmed" }
- takeScreenshot: checkout-done
$ skal test e2e                    # all flows, attached device/emulator
$ skal test e2e --target ios       # pick a platform; boots a simulator if needed
$ skal test e2e checkout           # one flow by name
[Passed] smoke (37s) · checkout (52s) — 2/2 flows
Match by id:, not visible text. Give every widget you assert or tap a testID — it becomes the platform accessibility identifier, survives copy changes, and is the same query your unit tests use. Skal widget labels live in the accessibility description, which Maestro's plain-string matcher doesn't read.

Debugging a failing flow

  • maestro hierarchy — dump exactly what Maestro sees on the current screen. If your testID isn't in there, the widget isn't in the semantics tree (or you're on the wrong screen).
  • maestro studio — point-and-click flow recorder in the browser; great for finding selectors.
  • On failure, Maestro saves screenshots + logs under ~/.maestro/tests/ — skal test e2e prints the path.
  • Very long screens: extremely tall single <ScrollView>s can defer semantics for their top content — prefer <ListView> (virtualized) for long pages, or assert on a shorter screen.

E2E in CI

The scaffold's GitHub Actions template runs flows on every PR — an Android emulator on a Linux runner, the app installed from the workflow's own release build:

# .github/workflows/e2e.yml (scaffolded)
- uses: reactivecircus/android-emulator-runner@v2
  with:
    api-level: 34
    script: |
      skal build android --debug
      adb install app-debug.apk
      skal test e2e --target android

Flows are the release gate: if a journey breaks, the merge waits — not your users.
